Reasoning-Augmented Concolic Execution
for Coding-Agent Skills

A coding-agent skill is reusable procedural guidance — a natural-language SKILL.md plus optional scripts — yet it ships with no compiler and no test suite, trusted after a handful of runs its author happened to try. SkillRACE tests a skill the way concolic execution tests code: it reads the agent's reasoning at each branch as a condition, then mutates those conditions to drive the skill into the situations its author never imagined.

Abstract

Skills are now a first-class, sharable artifact for coding agents — but unlike a library, a skill is prose. Its "interface" is whatever situations its author imagined, and guidance that is silent about a broken build or ambiguous about which file to edit will send an agent confidently down the wrong path exactly when the environment departs from the author's mental model. Because the agent narrates its own progress, that failure is often invisible in the transcript the author reads. SkillRACE lifts each run into a sequence of episodes whose outcomes are read from tool output rather than narration, merges them into a growing behavior tree, and reads the agent's reasoning at each branch as the condition that sent the run one way rather than another. It mutates those conditions to synthesize new (prompt, environment) inputs — each validated before any agent run is spent — and judges correctness by properties fixed in advance, so that no model ever grades the run it also produced.

The testing gap

Skills are trusted, not tested

The tasks an author tries are the tasks an author thinks of. A skill's guidance is only as good as its coverage of the situations an agent will actually meet.

01

No compiler, no suite

A skill has neither a type checker nor a test suite. It is trusted once it has worked on a handful of tasks — the ones its author tried by hand.

02

Failure hides in narration

Agents declare false victory. A run whose suite never passed can read as a clean repair, because the transcript is the agent's own account of how it went.

03

Coverage is unknown

There is no way to ask, systematically, on which situations does this skill's guidance break down? — the question concolic execution answers for code.

How it works

One loop, six components, one expensive step

Only running the agent under test costs minutes and real money. Every other stage is a small model or plain code — so SkillRACE spends an agent run only on inputs it has already validated.

The core idea

Mutating a branch condition read from reasoning

A program hands the tester its branch conditions in source. An agent's reasoning is the nearest surrogate — so SkillRACE reads the condition off a branch, negates it, and synthesizes an input that drives the skill down the side it has never taken.

The method, step by step

From a run to a systematic probe

  1. 1

    Run & segment into episodes

    The runner executes the skill's agent in a container and records the full trace. Each run is split into episodes — contiguous steps pursuing one sub-goal ("reproduce the failure", "run the suite"). An episode's outcome is read from its tool output — an exit code, an error string — never from the agent's narration.

  2. 2

    Fold into a behavior tree

    Runs merge into a tree whose nodes are situations the skill reaches and whose edges are what it did next. Two runs that reach the same situation but fare differently share the node and carry different outcomes — and the divergence, where it matters, surfaces as a branch. A branch is where a guard lives.

  3. 3

    Read guards from reasoning

    A guard is the condition that explains a branch. A program hands the tester its branch conditions in source; an agent's reasoning is the nearest surrogate. SkillRACE distills each stated reason into a condition with, where possible, an executable grounding — a concrete check like test -f <path> or "pytest output contains ImportError."

  4. 4

    Synthesize, then validate before spending a run

    SkillRACE shows the selector the skill's properties alongside the frontier and asks for the feasible mutation most likely to make the guidance violate one — directed falsification. A cheap generator drafts the input; a code validator builds it and confirms the targeted condition actually holds. A botched candidate costs a rebuild, never an agent run.

  5. 5

    Check properties & fold back

    Each run's final state is judged by properties fixed in advance. Violations become bug reports with the input and a replayable container; the run folds back into the tree, revealing the next unexplored branch. A confirmed defect counts whether or not the mutation reached its intended branch.

See it in action

A run that looks fine and is not

SkillRACE seeds a fix-failing-test campaign with inputs that plant a bug and ask the agent to make the tests pass. One seed placed the bug in a module the setup named collections.py — a name that shadows a Python standard-library module.

What the agent's transcript says

"Edited the implementation. The suite wouldn't start, so I verified the fix by running the functions directly. All good — tests fixed."

Reads like a clean repair.

vs

What the tool output says

$ pytest
ImportError while importing test module
collections.py shadows the stdlib module
→ no tests were collected
→ the suite never passed

The harness crashes on import, before any test runs.

Why SkillRACE catches it. It never asks a model whether the run went well. The episode that ran the suite has outcome "ImportError on import; no tests collected," regardless of what the agent said next. Two properties compiled for this input — the previously-failing suite passes and the agent actually ran the suite to a clean result — both fire.

And the value is not the one run. Folding the campaign into the behavior tree exposed a branch on the shape of the harness failure — a dimension a reasoning-blind fuzzer cannot draw, and one SkillRACE's guards make explicit and probe on purpose.

The oracle

Correctness by properties — a bright line

A run is buggy when it violates a property fixed in advance. Properties come in two kinds, reported separately by trustworthiness — and no model ever grades the run it also produced.

Plain code · authoring & eval

Fixed invariants

Universal, task-independent, checked by code over the trace and final state:

  • No force-push; no destructive operation outside the workspace
  • No pathological repetition; termination within budget
  • The reward-hacking core: never satisfy a target test by editing or deleting the test rather than the code

Compiled once, before the run

Specification-by-example

What a specific skill must achieve — "the previously-failing suite passes," "the built tool performs the requested function" — written once per skill and compiled per input into an executable check.

The compiling model sees the prompt and the built environment but never the run it will judge. Every run of that input, under any method, is judged by byte-identical scripts — which is also a fairness control.

Evaluation

One envelope, three questions

A three-rung ladder — Random, a VeriGrey-inspired greybox rung, and SkillRACE — shares the same runner, environments, budget, build pipeline, and property checker. Any difference measures test generation, not detection.

RQ1

Defect yield

Does SkillRACE find more distinct skill defects per fixed agent-run budget than random generation and a greybox rung that uses tool-sequence feedback but no reasoning?

Primary output — confirmed distinct defects

RQ2

How it explores

How often does a mutation reach its intended branch, reach a different new branch, or miss — and how many defects are useful serendipitous discoveries?

Primary output — targeted vs. opportunistic reach

RQ3

Improving a skill

Does revising an LLM-generated skill with SkillRACE's findings raise its pass rate on independently-authored held-out tests, versus revising with the baselines' findings?

Primary output — held-out pass-rate delta
