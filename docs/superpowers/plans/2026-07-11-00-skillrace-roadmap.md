# SkillRACE ISSTA Readiness Roadmap

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the approved evaluation design into a sequence of independently testable releases that ends with frozen, reproducible RQ1 and RQ3 experiments.

**Architecture:** Six plans follow the data and validity dependencies. Reliability and experimental accounting land before baseline changes; baseline semantics land before parallel execution; benchmark oracles land before RQ3 is run; analysis and protocol freezing are the final gate before expensive measurements.

**Tech Stack:** Python 3.12, pytest, Docker, Pi/CloseAI, Bash, JSON/JSONL, LaTeX.

## Non-negotiable experimental invariants

- [ ] Every method uses exactly the same configured agent model and total agent-execution budget.
- [ ] Independent random has no bootstrap corpus and generates a fresh test on every counted iteration.
- [ ] VeriGrey-inspired and SkillRACE use equal bootstrap counts, independently generated by the same frozen initialization protocol.
- [ ] Bootstrap executions count in yield curves and total budgets.
- [ ] The headline compares complete systems; the matched seeded black-box ablation isolates greybox feedback.
- [ ] Build/schema failures are reported generation failures and do not consume an agent-execution slot.
- [ ] Once Pi starts, timeout, agent error, and oracle-inconclusive runs consume a slot and remain in the manifest.
- [ ] A defect counts only after the frozen confirmation rule and method-blind failure-cause grouping.
- [ ] Intended branch reach is diagnostic, not an eligibility condition for a discovered defect.
- [ ] No campaign or reviser process can read RQ3 hidden tests.
- [ ] Full-scale result directories are created only from a hash-verified frozen protocol.

## Release checkpoints

### Checkpoint A: reliable sequential smoke

- [ ] Execute Plans 1 and 2.
- [ ] Run the offline suite.
- [ ] Run a 3-execution fake-agent campaign for all headline methods and the seeded no-feedback ablation.
- [ ] Verify random has zero bootstrap records; both adaptive methods have the configured count; all manifests resume without duplicate execution IDs.

### Checkpoint B: one-skill live pilot

- [ ] Execute Plan 3.
- [ ] Run one debugging, one CLI, one parser, one SQL, and one low-contingency skill with a small frozen budget.
- [ ] Verify immutable worker directories, deterministic fold order, resource limits, classification/fallback reporting, and no missing artifacts.
- [ ] Freeze the global VeriGrey event granularity on development skills only.

### Checkpoint C: one-scenario RQ3 dry run

- [ ] Execute Plans 4 and 5.
- [ ] Audit every hidden test against its reference, empty implementation, and targeted mutants.
- [ ] Run all six conditions and three repeats for one scenario.
- [ ] Verify that Phase 1 and the reviser were given a public-only staged directory with no `tests/` subtree.

### Checkpoint D: protocol freeze

- [ ] Execute Plan 6.
- [ ] Produce dataset, model, prompt, image, property, scenario, budget, and analysis hashes.
- [ ] Tag the frozen protocol before inspecting headline results.
- [ ] Run the sub-30-minute artifact smoke test from a clean checkout.

## Completion definition

The repository is ready for expensive ISSTA measurements only when every acceptance check in the approved design maps to a passing automated test or a committed, hash-verified protocol artifact. A favorable pilot result is not a completion criterion; trustworthy execution and frozen analysis are.
